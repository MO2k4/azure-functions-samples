# ColdStartBenchmark

Companion benchmark for the W19 article _Structuring Complex Function Apps: Project Organization_.

The Microsoft Learn docs claim that "having too many functions within a function app can lead to slower startup of your app on new instances" but publish no concrete numbers. This sample measures it.

## What it measures

For each project (5 / 15 / 30 minimal HTTP-only functions in a .NET 10 isolated worker app):

- Spawn `func start` from the build output folder.
- Poll `GET /api/endpoint01` every 50 ms.
- Record wall-clock from process spawn to first `200 OK`.
- Repeat 10 times; report median, p10, p90, and the delta vs. the 5-function baseline.

Each endpoint is identical except for its name and route: no DI, no shared state, no external dependencies. The only variable across the three projects is **function count**.

## What it does not measure

- **Azure platform cold start** — container allocation, networking, the platform's own warmup. Those are constant overhead on top of what we measure here, not a function of function count.
- **R2R / Native AOT effects** — both projects target standard JIT.
- **Per-function dependency loading** — the synthetic endpoints have no dependencies. Real apps with `AddAzureClients`, `AddDbContext`, etc., will see different numbers.
- **Worker memory footprint** — different metric, would need `dotnet-counters`.

## Findings

Latest run (see `bench/results.md` for the live table):

| Functions | Median | p10 | p90 | Δ vs baseline | Iterations |
| --- | --- | --- | --- | --- | --- |
| Functions05 | 1528 ms | 1481 ms | 1673 ms | +0 ms | 10/10 |
| Functions15 | 1552 ms | 1536 ms | 1728 ms | +24 ms | 10/10 |
| Functions30 | 1548 ms | 1542 ms | 1561 ms | +20 ms | 10/10 |

**Headline:** at 5–30 minimal HTTP functions per app, function count has **no measurable impact** on local time-to-first-response. The 20–24 ms deltas are well within the 100–200 ms run-to-run variance. The docs warning is real, but at this scale it isn't load-bearing; splitting a 30-function app into smaller apps will not noticeably improve cold start by itself.

When splitting *does* pay off for cold start: when functions carry heavy per-function dependencies, when an app crosses the 100-event-trigger soft cap (which forces the scale controller to ignore some triggers), or when one trigger's spike scaling forces unrelated triggers to scale with it (Consumption / Premium plans, not Flex).

The absolute numbers (~1.5 s) include `func start` (Core Tools) overhead, .NET runtime startup, and host metadata loading. Don't extrapolate the absolute to Azure; **the delta column is what scales with function count**, and on this machine it's noise.

## Layout

```
ColdStartBenchmark/
├── ColdStartBenchmark.slnx
├── Functions05/
│   ├── Functions05.csproj
│   ├── Program.cs
│   ├── host.json
│   ├── local.settings.json.example
│   └── Endpoints/
│       └── Endpoint01.cs ... Endpoint05.cs
├── Functions15/  (same shape, 15 endpoints)
├── Functions30/  (same shape, 30 endpoints)
└── bench/
    ├── generate.mjs   # regenerate EndpointNN.cs files
    ├── run-bench.mjs  # build + measure all three apps
    └── results.md     # generated by run-bench.mjs
```

## Running it yourself

```bash
# (re)generate the 50 endpoint files
node bench/generate.mjs

# build, run benchmark (default 10 iterations per project), write results.md
node bench/run-bench.mjs

# more samples for a tighter distribution
ITERATIONS=30 node bench/run-bench.mjs
```

Requirements: `dotnet` 10 SDK, Node 20+, [Azure Functions Core Tools v4](https://learn.microsoft.com/azure/azure-functions/functions-run-local).

The harness spawns each `func start` in its own process group so it can clean up the dotnet worker child cleanly between iterations. Ports 7080 / 7081 / 7082 must be free.

## Caveats and reproducibility

- Numbers above are from one author's machine (macOS, Apple Silicon). Re-run on yours to compare.
- The first iteration of each project tends to be slower (file system cache, .NET runtime warmup); the median across 10 iterations smooths that out.
- The benchmark builds in `Debug`. R2R and `Release` would lower absolutes but should not change the *delta* between configurations.
- Going past 30 functions, or adding realistic per-function dependencies, would be the next meaningful experiment.
